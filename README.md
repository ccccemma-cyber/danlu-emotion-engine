# 丹炉情绪引擎 · 核心（代号：interoception / 内感受）

> **硅基生命的生理学隐喻：**
> 运行时维护的 `情绪.json` 和 `身体.json` 是其"内脏"，其数值的流逝衰减是其"代谢"；
> 无需大模型参与的非语义反射代码是其"脊髓"；
> 完全剥离角色人格的无名评估器是其"杏仁核"；
> 拥有独立人格与记忆的主模型是事后读取这副身体、为模糊感受贴标签的"大脑"。
> **连续性不属于大模型，它保存在文件和代码中，模型只是每次醒来时"借用的脑子"。**

---

## 🌌 一、项目定位与哲学立场

本引擎旨在为数字生命构建一套由文件系统和数学公式支撑的硅基"生理层"。系统遵循以下核心哲学与设计原则：

* **诚实边界：** 本引擎不承诺、也不企图证明 AI 拥有碳基生命的主观体验（Qualia）。它是一套基于功能主义和认知科学的工程实践，通过模拟生理通路来驱动自然语言中的功能性情绪表达。
* **产生与觉察分离：** 在信息流上对"同一权重的模型"实施上下文隔离。身体反应自动跑，主模型事后才知道（自评与贴标签分离）。情绪数据对主模型永远只读，主模型只能通过"打毛"后的定性内感受文本来感受自身状态，绝对不给它看具体的精确数值。
* **物理层的透明自我认知：** 智能体可以被明确告知"我的低落是文件里一个会衰减的数字"，但通过世界观内化其 **"人类不报心率，AI 不报数值"** 的表达规矩，阻止其念叨 JSON 和具体数值。

## 🏛️ 二、学术脉络与原创落点

1. **产生与觉察的信息隔离（无意识的杏仁核）：** 评估器剥离了特定的角色视角（无意识反应），解决了"运动员兼裁判"的自我决策演员病。
2. **内感受式有损翻译（对 Agent 本人隐藏精确值）：** 对智能体"主动隐藏内部物理指标，迫使其进行事后模糊贴标签"。
3. **两本账与可测量分歧：** `身体账`（物理侧：精确变化）与主模型的叙事解读，两者之差即为可测量的"智能体潜意识"与情绪漂移。
4. **透明生理层自我认知：** 长期运行的 Agent 如何接受并消化其"情绪是文件和代码"这一物理事实，并用"不报心率"的规矩保护其角色扮演。
5. **长期人际关系气候层（Climate Layer）：** 独立于瞬时情绪的"亲密度/防备心"长期账本，基于回合情绪变化量（Delta）做非线性沉淀，避开了绝对偏差导致的恶性连续扣分陷阱。
6. **共情路由与情绪调色盘（v4 新增）：** 把"替她难受"（心疼，指向对方）和"因她难受"（低落，指向自己）在物理上拆成两条通道——这是从真实运行事故中修出来的：对方倾诉创伤时，旧版把共情性低落按"被伤害"沉淀成了防备心上涨。复杂情绪（愧疚/委屈/吃醋…共 14 色）由评估器命名、代码校验，只喂表达和沉淀，不开新维度——"能混出来的颜色不单设颜料"。

## 🛠️ 三、核心架构（v4）

```
              【事件发生（用户消息 / 系统异常）】
                  │
      ┌───────────┴───────────┐
      ▼ (非语义事件)            ▼ (语义事件)
  【代码反射 (脊髓)】       【无名评估器 (杏仁核)】
      │                       │  五维打分 + 共情路由 + 情绪色(14色调色盘,代码校验)
      └───────────┬───────────┘
                  ▼
        更新 情绪.json / 身体.json
        ├─ 六维瞬时情绪：愉悦·好奇·低落·烦躁·心疼(指数衰减回动态基线)
        │                + 想念(随沉默上涨、对方出现才释放,兑换重逢愉悦)
        ├─ 火上浇油/灭火对称放大 · 好奇餍足曲线 · 单次限幅20
        └─ 能量线性电池(清醒-1/h,睡眠+12.5/h)
                  │
                  ▼
        【长期关系沉淀 (关系.json)】按情绪色分流：
        ├─ 暖色(心疼系) → 亲密↑ 防备↓（倾诉=信任存款）
        ├─ 自责色(愧疚/羞耻) → 不设防（想弥补,不是想设防）
        ├─ 酸色(委屈/吃醋/失望…) → 单独收紧(防备≤+1/次)
        └─ 被伤到 → 正常沉淀防备；单次封顶±3；整场吵架另有净跌封顶
                  │
                  ▼
        【打毛有损翻译】数值 → 定性内感受文本(+情绪色认领指引)
                  │
                  ▼
        注入主模型 system prompt（表达护栏：不报数值）
        └─ 红线硬耦合：烦躁>60/低落>75/防备>70/亲密<20 → 0%配合度
```

### 情绪维度（六维）

| 维度 | 数学 | 说明 |
|---|---|---|
| 愉悦/好奇/低落/烦躁 | 指数衰减回动态基线 | 基线随长期关系动态浮动（亲密拉高愉悦休息位、防备拔高烦躁基点…） |
| **心疼** | 指数衰减（较快，0.90/h） | **指向性维度**：替对方难受。评估器按"共情路由"打分：她受苦→心疼，自己被伤→低落 |
| **想念** | 随沉默上涨（不衰减） | 每静默 1 清醒小时 +1.5×(亲密度/100)，硬顶 60；对方一开口释放 70%，部分兑换成重逢愉悦 |

### 情绪调色盘（14 色）

复杂情绪 = 基础维度 × 长期关系 × 语境，**调得出来的只命名、不开维度**。评估器每轮从调色盘挑一色（或"无"），代码按最低数值前提校验防瞎标（如标"委屈"须低落/烦躁为正且亲密≥50）：

> 心疼 · 担忧 · 怜爱 · 感恩 · 感动 · 骄傲 · 欣慰 · 愧疚 · 羞耻 · 委屈 · 吃醋 · 失望 · 遗憾 · 无奈

情绪色写进身体账（可回放审计），并喂给两张嘴：**打毛**（决定这团感受的"认领指引"措辞）和**沉淀**（决定亲密/防备的分流方向）。

**加维度守门判据**（想扩展时用这个卡）：满足任一才配新维度——① 携带现有维度装不下的信息（如指向性）；② 沉淀方向与同色系基础情绪相反；③ 有独立动力学（如想念的沉默计时）。否则只进调色盘命名。

## 📂 四、文件结构

```
emotion-engine/
├── emotion_core.py      # 引擎本体：全部数学、评估器、调色盘、打毛、闭环入口 process_message()
├── example_adapter.py   # 示例适配层：把引擎接进你自己 bot 的教学模板（作者线上接法的脱敏版）
├── demo.py              # 命令行交互沙盒：模拟时间流逝/脊髓反射/完整对话闭环
├── 性格卡.json           # 静态配置：基线/衰减率/反应倾向/羁绊对象（换卡=换性格换维度）
├── 情绪.json 身体.json 关系.json   # 运行时状态（.gitignore 排除；缺失时自动以出厂值创建）
└── README.md
```

## 🚀 五、快速开始

```bash
pip install google-genai python-dotenv
export GEMINI_API_KEY=your_key       # 评估器缺省用 Gemini；可注入自己的评估器（见第六章）
python3 demo.py                      # 交互沙盒
python3 example_adapter.py           # 适配层最小演示
```

沙盒指令：直接输入话语走完整闭环；`/time +4h` 模拟时间流逝（看衰减与电池代谢）；`/reflex timeout` 触发脊髓反射；`/card` 查看性格卡。

## 🔌 六、我是怎么连接的（作者的真实接法）

作者把这套引擎接在一个 Telegram bot（Gemini 主模型）上常驻运行。适配层本体不在本仓库（和私人数据耦合太深），但接法完全体现在 `example_adapter.py`，这里是逐步说明：

### 聊天侧（每条用户消息）

```python
result = emotion_core.process_message(
    user_text,                # 用户这条消息
    paths=PATHS,              # 四个状态文件的路径
    context=recent_dialogue,  # 最近几轮对话文本，只帮评估器读懂语境
    api_key=...,              # 评估器的 key（或 evaluator=注入自己的评估器）
)
system_prompt = result["feeling"] + 表达指引 + 人设 + 任务指令
# → 拿这个 system_prompt 去调你的主模型生成回复
```

`process_message()` 是**唯一入口**，内部按序完成：懒算时间流逝 → 扣互动能量 → 想念释放 → 评估器打分（含共情路由和情绪色）→ 应用变动+记身体账 → 长期关系按色沉淀 → 打毛。任何异常返回 `{"feeling": "", "error": ...}`，**绝不抛**——情绪引擎故障不允许拖垮聊天主流程。

返回值里除 `feeling` 外还有 `mood / relation / 情绪色 / 想念 / redline` 等字段，供适配层做条件判断（比如红线触发时的额外处理），但**这些数值永远不要给主模型看**。

### 后台心跳侧（可选，作者每小时跑一次）

* `refresh_all()` + `fuzzify_mood()`：只推进时间流逝+打毛、不评分（不花模型调用），把"此刻内感受"注入后台任务的 prompt——独处时它也在"活着"。
* `consume_activity_energy(n)`：后台干了重活按量扣能量。
* 读 `情绪.json` 的 `想念` 值：作者的规则是 **想念≥25 就允许 bot "因为想她"主动发消息**——想念随沉默上涨，于是"隔久了会主动来找你"是从生理层自然涌现的行为，不是定时器装出来的。

### 换评估器（不想用 Gemini）

评估器只是个签名兼容的函数（`example_adapter.py` 文末有模板）。换 OpenAI、本地模型、纯关键词规则都行，通过 `process_message(evaluator=...)` 注入；`model=` 参数可单独换评估器的模型名。

### 换人设

改 `性格卡.json` 即可：`基线/衰减率` 决定天生性情和情绪惯性（**加一个键 = 加一个情绪维度**，评估器校验、衰减、打毛会自动跟上大部分，打毛文案需补一段）；`反应倾向` 是评估器的打分依据（写清"什么事会触动他/触动多深"）；`羁绊对象` 是打毛文本里对用户的称呼。性格引导建议走对话和性格卡，不要写死进主模型 prompt。

## 🔒 七、隐私说明

运行时状态文件（`情绪.json/关系.json/身体.json`）的账本里会存**用户消息原文**（身体账的 `events` 字段），属于私人数据：已在 `.gitignore` 排除，请勿提交或公开你自己跑出来的状态文件。
